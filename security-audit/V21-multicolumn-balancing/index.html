V21 · Superlinear multicolumn balancing in content size

Medium   Denial of service (CPU)

Status fixed
Key file source/layout/multicolumnbox.cpp

Nature

Diagnosis corrected by temporary instrumentation (per-line and per-pass timing): the balancing loop (MultiColumnFlowBox::layoutContents(), while(changed) {{ ...; changed = layoutColumns(true); }}) is not the cause of the superlinear cost observed on the PoC. The initial column-height estimate (distributeImplicitBreaks() / calculateColumnHeight(false)) is exactly totalHeight / columnCount whenever there is no explicit column break -- verified both algebraically and empirically across many content shapes (uniform paragraph, hundreds of blocks of random heights, dozens of column-span:all): the loop consistently converges in 1 or 2 iterations, regardless of content size. The real superlinear cost (confirmed by per-line timing: the cost per line grows with position in the document, roughly 4 to 5x more expensive around line 4000 than around line 200 of the same document) comes from TextShapeRun::positionForOffset() / offsetForPosition() (source/graphics/textshape.cpp, outside this file): these functions rescan the glyph array from index 0 on every call instead of resuming from a cached position, which makes each new line of a long text run O(position) and the whole document O(n²) -- reproduces identically on plain text with no columns (same scaling measured on normal paginated content). This is therefore a cost shared by any long text layout, distinct from multicolumn balancing and out of scope for this fix; flagged as a separate follow-up. What remains real and specific to this file: nothing in the balancing loop's exit condition formally guarantees fast termination -- converging in 1-2 iterations is an empirical/algebraic fact for the content shapes tested, not a proven property for every future adversarial input.

Location

  • source/layout/multicolumnbox.cpp — MultiColumnFlowBox::layoutContents(): balancing loop with no iteration bound
  • include/plutobook.hpp — EngineLimits::maxColumnBalancingIterations

Risk & impact

Impact: large multicolumn content inherits a superlinear text-layout cost (out of scope, see Nature); independently, the balancing loop itself had no explicit bound on its number of iterations.

Reproduction

  1. Generate poc/make-multicol.py (large content, columns:3) → render → slow both before and after this fix (the real cause is out of scope, see Nature); the iteration cap is verified separately via a dedicated C harness (an artificially low cap = measurable early convergence and still-complete PDF; the default cap = byte-identical to unlimited).

Example files:

  • poc/make-multicol.py

Proposed fix

Explicitly documented fallback: the "clean algorithmic fix" alternative (binary-search the column height in O(log) iterations instead of the current increment-by-minimum-shortage) was dropped because the current mechanism turned out to already be near-optimal empirically/algebraically for this file -- the real performance win would require fixing TextShapeRun::positionForOffset / offsetForPosition, out of scope for this fix (see Nature). New EngineLimits::maxColumnBalancingIterations limit: MultiColumnFlowBox::layoutContents() stops refining the balancing once this many extra relayout passes have been reached without convergence -- columns may then be slightly less perfectly balanced than an unbounded search would produce, but layout stays correct and terminates.

Configuration (default + configurable)

New EngineLimits::maxColumnBalancingIterations limit (setMaxColumnBalancingIterations / maxColumnBalancingIterations()), default 10 (comfortably above the 1-2 iterations observed across every content shape tested, both legitimate and adversarial), 0 = unlimited (not recommended). C API plutobook_set_max_column_balancing_iterations(unsigned int). Verified: (a) non-regression -- normal multicolumn documents (2 and 3 columns, column-gap, column-span:all) render byte-identical before/after (pre-fix binary via git stash + separate build); (b) the default cap (10) produces a byte-identical PDF to the unlimited cap (0) on content with many spanners, confirming it changes nothing for real content; (c) a deliberately low cap (1, via a dedicated C harness calling plutobook_set_max_column_balancing_iterations) produces a different PDF (the mechanism stops much earlier) but still complete and valid -- all content is present (100/100 markers verified on both sides), only column balance differs, confirming the cap works and terminates cleanly with no content loss. Caveat: this fix does not make the literal PoC converge to linear time -- the real cause is out of scope (see Nature).

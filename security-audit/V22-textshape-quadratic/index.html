V22 · TextShapeRun::positionForOffset/offsetForPosition are O(n²) per run

High   Denial of service (CPU)

Status open
Key file source/graphics/textshape.cpp

Nature

Discovered while diagnosing V21. TextShapeRun::positionForOffset() and offsetForPosition() rescan the glyph array from index 0 on every call. Line layout calls them once per line: for a long text run, each new line costs O(position) and the whole run becomes O(n²). Reproduces on plain paginated single-column text (independent of multicolumn): this is the real cause of the slowdown seen in V21's PoC.

Location

  • source/graphics/textshape.cpp — positionForOffset/offsetForPosition rescan the glyph array from index 0 on every call
  • source/layout/linelayout.cpp — called once per line during layout -> O(position) per line

Risk & impact

Impact: CPU hang/slowness (DoS) from a long text document (multicolumn or not). ~4-5x cost per line late in a document vs. early, measured.

Reproduction

  1. Generate poc/make-longtext.py (long single-column text), render: superlinear pre-fix, ~linear post-fix.
  2. V21's multicolumn PoC also becomes bounded after this fix.

Example files:

  • poc/make-longtext.py

Proposed fix

Don't rescan from 0: resume from a cached cursor, or binary-search the (monotonic) glyph offsets, making the run O(n) (or O(n log n)).

Configuration (default + configurable)

Algorithmic fix (no knob; text-node length is already bounded by V19).

V22 · TextShapeRun::positionForOffset/offsetForPosition are O(n²) per run

High   Denial of service (CPU)

Status fixed (f8e4389)
Key file source/graphics/textshape.cpp

Nature

Discovered while diagnosing V21. TextShapeRun::positionForOffset() and offsetForPosition() rescan the glyph array from index 0 on every call. Line layout calls them once per line: for a long text run, each new line costs O(position) and the whole run becomes O(n²). Reproduces on plain paginated single-column text (independent of multicolumn).

Diagnosis refined while fixing: fixing only these two functions (binary search instead of the rescan) was not enough -- backed by empirical measurement (5000/10000/20000/40000 words: 0.8s / 2.9s / 11.4s / 42s, still quadratic). Instrumentation via iteration counters: the real dominant chunk was in TextShapeView::width(), maxAscentAndDescent(), and draw() (and, by the same pattern, expansionOpportunityCount()) -- these methods walk every run of the shape on every call and, for any run entirely outside the current view's [startOffset,endOffset) range, still scan the entirety of its glyphs to zero effect (no condition ever becomes true), purely to "skip" it. Same situation as a single long run: viewing a late line revisits everything before it on every call. Once these four methods were fixed in addition to the two named in the title, the measured scaling becomes near-linear again (see Fix).

Location

  • source/graphics/textshape.cpp — positionForOffset/offsetForPosition rescan the glyph array from index 0 on every call
  • source/graphics/textshape.cpp — TextShapeView::width/maxAscentAndDescent/draw/expansionOpportunityCount also rescan every run from glyph 0
  • source/layout/linelayout.cpp — called once per line during layout -> O(position) per line

Risk & impact

Impact: CPU hang/slowness (DoS) from a long text document (multicolumn or not). ~4-5x cost per line late in a document vs. early, measured; without the complete fix (the 4 TextShapeView methods included), the 200,000-word PoC still times out (>40s) even with the binary search applied to just the two functions named in this finding's title.

Reproduction

  1. Generate poc/make-longtext.py (long single-column text), render: superlinear pre-fix (40s+ timeout), ~linear post-fix (1.4s).
  2. V21's multicolumn PoC also becomes fast after this fix (60s timeout pre-fix -> 0.24s post-fix, measured on V21's poc/make-multicol.py).

Example files:

  • poc/make-longtext.py

Proposed fix

Binary search on (monotonic) glyph offsets instead of a rescan from 0: TextShapeRun::positionForOffset()/offsetForPosition() become O(log n) per call via a new private glyphIndexForOffset() method and a cumulative advance sum (advanceUpTo) precomputed once when the run is created (O(n), amortized over every later call). positionForOffset searches on characterIndex, monotonic by HarfBuzz construction (ascending for LTR, descending for RTL) regardless of advance sign -- always safe. offsetForPosition searches on the cumulative sum itself, which is only monotonic if every advance is ≥ 0 (the normal case); a sufficiently strong negative letter-spacing/word-spacing can make an advance negative -- in that case (rare, detected once per run at creation), offsetForPosition falls back to the original linear scan verbatim, to guarantee an identical result under every circumstance rather than assuming a monotonicity that isn't guaranteed.

Necessary complement (see Nature): TextShapeView::width(), maxAscentAndDescent(), draw(), and expansionOpportunityCount() fixed for the same ailment with the same technique -- a new rangeForView() function skips any run entirely outside the view in O(1) (and stops the outer loop as soon as the remaining runs are necessarily out of range too, since run order is ascending in LTR / descending in RTL) and uses glyphIndexForOffset() to jump straight to the first relevant glyph of the run overlapping the view, instead of rescanning from 0. Special case for draw(): it has a visible per-run side effect (cairo calls), so unlike the other three (pure, no effect for a skipped run), its outer loop still visits every run in the same order and issues the same cairo calls for each (only the glyph scan itself is shortened/skipped, which leaves numGlyphs at 0 exactly as the original scan would for such a run).

Configuration (default + configurable)

Algorithmic fix (no knob; text-node length is already bounded by V19). Verification -- scaling: poc/make-longtext.py (200k words) 40s+ timeout → 1.4s; at 2x (400k words) 3.7s, at 4x (800k words) 11.0s (mildly superlinear beyond the textshape.cpp fix itself -- confirmed by iteration counters that the fixed TextShapeView and TextShapeRun methods scale perfectly linearly with document size; the residual comes from elsewhere in the pipeline, out of scope); 5000/10000/20000/40000 words go from 0.8s/2.9s/11.4s/42s (quadratic, partial fix) to 0.075s/0.098s/0.15s/0.27s (near-linear, full fix). V21 PoC (poc/make-multicol.py, 20000 words / 3 columns): 60s timeout pre-fix (binary via git stash + separate build) → 0.24s post-fix. Non-regression, byte-identical (pre-fix binary via git stash + separate meson build, compared to the post-fix binary) on 14 documents: plain LTR multi-paragraph, justified text (expansionOpportunityCount), mixed fonts/styles/sizes within one line, ligature-heavy words (ffi/ffl/fi/fl), negative letter-spacing (fallback path included), Arabic RTL (Noto Sans Arabic), Hebrew RTL (Noto Serif Hebrew), mixed LTR/RTL/RTL bidi paragraph in one line, tabs, edge cases (empty/single-character/single-space paragraph), SVG text (TextShapeView::draw path, distinct from linelayout.cpp), reduced longtext, reduced V21 multicolumn PoC -- all PDF output byte-identical (cmp); 6 of these documents also byte-identical as PNG output (html2png).

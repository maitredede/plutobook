V22 · TextShapeRun::positionForOffset/offsetForPosition en O(n²) par run

Haute   Deni de service (CPU)

Statut corrige
Fichier cle source/graphics/textshape.cpp

Nature

Decouvert en diagnostiquant V21. TextShapeRun::positionForOffset() et offsetForPosition() reparcourent le tableau de glyphes depuis l'index 0 a chaque appel. Le line layout les appelle une fois par ligne : pour un long run de texte, chaque nouvelle ligne coute O(position) et le run entier devient O(n²). Se reproduit sur du texte simple colonne pagine (independant du multicolonne).

Diagnostic affine pendant le correctif : corriger uniquement ces deux fonctions (recherche dichotomique au lieu du rescan) n'a pas suffi -- mesure empirique a l'appui (5000/10000/20000/40000 mots : 0,8s / 2,9s / 11,4s / 42s, toujours quadratique). Instrumentation par compteurs d'iterations : le veritable morceau dominant etait dans TextShapeView::width(), maxAscentAndDescent() et draw() (et par le meme motif, expansionOpportunityCount()) -- ces methodes parcourent tous les runs de la shape a chaque appel et, pour tout run entierement hors de la plage [startOffset,endOffset) de la vue courante, balaient neanmoins la totalite de ses glyphes sans le moindre effet (aucune condition ne devient vraie), uniquement pour le "sauter". Meme situation qu'un run unique assez long : la vue d'une ligne tardive revisite tout ce qui precede a chaque appel. Une fois ces quatre methodes corrigees en plus des deux nommees dans le titre, la mise a l'echelle mesuree redevient quasi lineaire (voir Correctif).

Localisation

  • source/graphics/textshape.cpp — positionForOffset/offsetForPosition rescannent les glyphes depuis l'index 0 a chaque appel
  • source/graphics/textshape.cpp — TextShapeView::width/maxAscentAndDescent/draw/expansionOpportunityCount rescannent aussi tous les runs depuis le glyphe 0
  • source/layout/linelayout.cpp — appelees par ligne lors du layout -> O(position) par ligne

Risque & impact

Impact : hang/lenteur CPU (DoS) a partir d'un long document texte (multicolonne ou non). Cout ~4-5x par ligne en fin de document vs debut, mesure ; sans le correctif complet (les 4 methodes de TextShapeView incluses), le PoC 200 000 mots reste hors delai (>40s) meme avec la recherche dichotomique sur les deux seules fonctions nommees dans ce finding.

Reproduction

  1. Generer poc/make-longtext.py (long texte simple colonne), rendre : superlineaire pre-fix (40s+ timeout), ~lineaire post-fix (1,4s).
  2. Le PoC multicolonne de V21 devient egalement rapide apres ce correctif (60s timeout pre-fix -> 0,24s post-fix, mesure sur poc/make-multicol.py de V21).

Correctif propose

Recherche dichotomique sur les offsets de glyphes (monotones) au lieu d'un rescan depuis 0 : TextShapeRun::positionForOffset()/offsetForPosition() deviennent O(log n) par appel via une nouvelle methode privee glyphIndexForOffset() et une somme cumulee des avances (advanceUpTo) precalculee une seule fois a la creation du run (O(n), amorti sur tous les appels ulterieurs). positionForOffset recherche sur characterIndex, monotone par construction HarfBuzz (croissant pour LTR, decroissant pour RTL) quel que soit le signe des avances -- toujours sur. offsetForPosition recherche sur la somme cumulee elle-meme, qui n'est monotone que si toutes les avances sont ≥ 0 (le cas normal) ; un letter-spacing/ word-spacing negatif suffisamment fort peut rendre une avance negative -- dans ce cas (rare, detecte une fois par run a la creation), offsetForPosition retombe sur le rescan lineaire original verbatim, pour garantir un resultat identique en toute circonstance plutot que de supposer une monotonie non garantie.

Complement necessaire (voir Nature) : TextShapeView::width(), maxAscentAndDescent(), draw() et expansionOpportunityCount() corrigees du meme mal avec la meme technique -- une nouvelle fonction rangeForView() saute en O(1) tout run entierement hors de la vue (et arrete la boucle externe des que les runs restants le sont necessairement aussi, l'ordre des runs etant croissant en LTR / decroissant en RTL) et utilise glyphIndexForOffset() pour sauter directement au premier glyphe pertinent du run qui chevauche la vue, au lieu de rescanner depuis 0. Cas particulier de draw() : elle a un effet de bord visible par run (appels cairo), donc contrairement aux trois autres (pures, sans effet pour un run saute), sa boucle externe continue de visiter tous les runs dans le meme ordre et emet les memes appels cairo pour chacun (seul le balayage de glyphes est raccourci/supprime, ce qui laisse numGlyphs a 0 exactement comme le balayage original l'aurait fait pour un tel run).

Configuration (defaut + configurable)

Correctif algorithmique (pas de knob ; la longueur de noeud texte est deja bornee en V19). Verification -- mise a l'echelle : poc/make-longtext.py (200k mots) 40s+ timeout → 1,4s ; a 2x (400k mots) 3,7s, a 4x (800k mots) 11,0s (legerement superlineaire au-dela de la correction textshape.cpp elle-meme -- confirme par compteurs d'iterations que les methodes TextShapeView et TextShapeRun corrigees scalent, elles, parfaitement lineairement avec la taille du document ; le residu vient d'ailleurs dans le pipeline, hors perimetre) ; 5000/10000/20000/40000 mots passent de 0,8s/2,9s/11,4s/42s (quadratique, correctif partiel) a 0,075s/0,098s/0,15s/0,27s (quasi lineaire, correctif complet). PoC V21 (poc/make-multicol.py, 20000 mots / 3 colonnes) : 60s timeout pre-fix (binaire via git stash + build separe) → 0,24s post-fix. Non-regression, identite au byte pres (binaire pre-fix via git stash + build meson separe, compare au binaire post-fix) sur 14 documents : LTR multi-paragraphe, texte justifie (expansionOpportunityCount), polices/styles/tailles melanges dans une meme ligne, mots a ligatures (ffi/ffl/fi/fl), letter-spacing negatif (y compris chemin de repli), arabe RTL (Noto Sans Arabic), hebreu RTL (Noto Serif Hebrew), paragraphe bidi LTR/RTL/RTL melange dans la meme ligne, tabulations, cas limites (paragraphe vide/un caractere/un espace), texte SVG (chemin TextShapeView::draw distinct de linelayout.cpp), longtext reduit, PoC V21 multicolonne reduit -- toutes les sorties PDF identiques au byte pres (cmp) ; 6 de ces documents egalement identiques au byte pres en sortie PNG (html2png).

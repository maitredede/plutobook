V22 · TextShapeRun::positionForOffset/offsetForPosition en O(n²) par run

Haute   Deni de service (CPU)

Statut a corriger
Fichier cle source/graphics/textshape.cpp

Nature

Decouvert en diagnostiquant V21. TextShapeRun::positionForOffset() et offsetForPosition() reparcourent le tableau de glyphes depuis l'index 0 a chaque appel. Le line layout les appelle une fois par ligne : pour un long run de texte, chaque nouvelle ligne coute O(position) et le run entier devient O(n²). Se reproduit sur du texte simple colonne pagine (independant du multicolonne) : c'est la vraie cause du ralentissement du PoC de V21.

Localisation

  • source/graphics/textshape.cpp — positionForOffset/offsetForPosition rescannent les glyphes depuis l'index 0 a chaque appel
  • source/layout/linelayout.cpp — appelees par ligne lors du layout -> O(position) par ligne

Risque & impact

Impact : hang/lenteur CPU (DoS) a partir d'un long document texte (multicolonne ou non). Cout ~4-5x par ligne en fin de document vs debut, mesure.

Reproduction

  1. Generer poc/make-longtext.py (long texte simple colonne), rendre : superlineaire pre-fix, ~lineaire post-fix.
  2. Le PoC multicolonne de V21 devient egalement borne apres ce correctif.

Fichiers d'exemple :

  • poc/make-longtext.py

Correctif propose

Ne pas rescanner depuis 0 : reprendre depuis un curseur memorise, ou faire une recherche dichotomique sur les offsets de glyphes (monotones), rendant le run O(n) (ou O(n log n)).

Configuration (defaut + configurable)

Correctif algorithmique (pas de knob ; la longueur de noeud texte est deja bornee en V19).

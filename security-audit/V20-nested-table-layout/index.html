V20 · Layout de tables imbriquees a cout exponentiel

Haute   Deni de service (CPU)

Statut corrige (5e2f11e)
Fichier cle source/layout/tablebox.cpp

Nature

Diagnostic confirme par comptage d'appels (instrumentation temporaire) : le cache de largeur preferee min/max par boite (BoxFrame::m_minPreferredWidth/m_maxPreferredWidth, invalide seulement si < 0) fonctionne correctement -- minPreferredWidth()/maxPreferredWidth() ne sont calcules qu'une fois par boite, cout O(N) mesure pour N tables imbriquees. Le veritable doublement vient du layout (pas du calcul de largeur) : l'algorithme de hauteur de table CSS 2.1 est intrinsequement a deux passes par table -- TableSectionBox::layout() relaie d'abord chaque cellule a sa hauteur naturelle (non etiree) pour mesurer la hauteur de chaque ligne, puis TableSectionBox::layoutRows() relaie a nouveau chaque cellule avec cette hauteur de ligne appliquee en override, pour etirer les cellules a la ligne (necessaire pour vertical-align et les descendants en hauteur pourcentage). Les deux passes relaient integralement le contenu de la cellule, y compris une table imbriquee a l'interieur : une table imbriquee sur N niveaux est donc laid out ~2^N fois, pas N fois -- mesure : 62, 2046, 65534 appels de layout de cellule pour N=5, 10, 15 (formule exacte 2^(N+1)-2), independant du cap de profondeur d'imbrication generale de V08 (512).

Localisation

  • source/layout/tablebox.cpp — TableSectionBox::layout() / layoutRows(): layout complet de chaque cellule, deux fois par niveau
  • source/layout/tablebox.cpp — TableBox::build() / exceedsMaxNestingDepth(): profondeur d'imbrication et coupe-circuit
  • include/plutobook.hpp — EngineLimits::maxTableNestingDepth

Risque & impact

Impact : hang CPU (DoS) sous le cap de profondeur general.

Reproduction

  1. Generer poc/make-nested-tables.py (~150 niveaux) → rendre → hang pre-fix (>20s), ~0.9s post-fix.

Fichiers d'exemple :

  • poc/make-nested-tables.py

Correctif propose

La memoisation seule ne corrige pas le probleme (le calcul de largeur est deja mis en cache et lineaire ; le cout exponentiel vient du layout, pas du calcul de largeur, et memoiser un layout complet -- positions, overflow, fragmentation -- serait bien plus invasif et risque qu'une largeur). Correctif choisi (repli explicitement documente dans ce dossier comme alternative acceptable) : borner specifiquement la profondeur d'imbrication des tables (distincte du cap general de 512). TableBox connait desormais sa profondeur parmi ses ancetres tables (m_nestingDepth, derivee dans build() -- O(1) amorti, l'ancetre le plus proche a deja sa propre profondeur calculee puisque build() parcourt l'arbre du haut vers le bas). Au-dela de EngineLimits::maxTableNestingDepth(), TableSectionBox::layoutRows() saute la seconde passe (etirement) pour les cellules de cette table : la cellule garde la hauteur naturelle mesuree par la premiere passe -- meme contenu, meme largeur, meme position, simplement pas etiree si une cellule voisine de la meme ligne est plus haute. Ce doublement recursif fait que le cout total est O(2^limite × profondeur_totale) et pas seulement O(2^limite) (chaque niveau au-dela de la limite dedouble a nouveau tout ce qui est imbrique au-dessus) : la limite doit donc rester basse pour que 2^limite reste une petite constante, contrairement aux autres limites de ce tableau ou une valeur de l'ordre de 100 suffirait.

Configuration (defaut + configurable)

Nouvelle limite EngineLimits::maxTableNestingDepth (setMaxTableNestingDepth/maxTableNestingDepth()), defaut 8, 0 = illimite (non recommande, cout reellement exponentiel sans elle). API C plutobook_set_max_table_nesting_depth(unsigned int). Verifie : mise a l'echelle desormais lineaire (0.19s/0.63s/0.89s/0.98s pour 50/100/150/200 niveaux, contre hang/timeout pre-fix des N≈20-25) ; PoC 150 niveaux -> ~0.92s, PDF valide (pre-fix : timeout >20s). Non-regression verifiee par comparaison PDF octet-par-octet entre binaire pre-fix (git stash + build separe) et post-fix sur : grille simple, colspan/rowspan, imbrication a 2 niveaux, largeurs %/fixe/auto, caption (les 5 -- identiques au byte pres) ; imbrication reelle jusqu'a exactement la limite (profondeur 8, incluse) -- identique au byte pres ; au-dela de la limite (profondeur 9+), differences observees uniquement quand une ligne contient des cellules de hauteurs naturelles differentes (la plus courte n'est plus etiree) -- comportement exactement documente, pas de corruption.

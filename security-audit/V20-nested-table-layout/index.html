V20 · Exponential-cost nested table layout

High   Denial of service (CPU)

Status fixed
Key file source/layout/tablebox.cpp

Nature

Diagnosis confirmed by call counting (temporary instrumentation): the per-box min/max preferred-width cache (BoxFrame::m_minPreferredWidth/m_maxPreferredWidth, invalidated only if < 0) works correctly -- minPreferredWidth()/maxPreferredWidth() are computed only once per box, an O(N) cost measured for N nested tables. The real doubling comes from layout (not the width computation): the CSS 2.1 table height algorithm is inherently two passes per table -- TableSectionBox::layout() first lays out each cell at its natural (unstretched) height to measure each row's height, then TableSectionBox::layoutRows() lays out every cell again with that row height applied as an override, to stretch cells to the row (needed for vertical-align and percentage-height descendants). Both passes fully lay out the cell's content, including any table nested inside it: a table nested N levels deep is therefore laid out ~2^N times, not N times -- measured: 62, 2046, 65534 cell-layout calls for N=5, 10, 15 (exact formula 2^(N+1)-2), independent of V08's general nesting-depth cap (512).

Location

  • source/layout/tablebox.cpp — TableSectionBox::layout() / layoutRows(): full layout of every cell, twice per level
  • source/layout/tablebox.cpp — TableBox::build() / exceedsMaxNestingDepth(): nesting depth and circuit breaker
  • include/plutobook.hpp — EngineLimits::maxTableNestingDepth

Risk & impact

Impact: CPU hang (DoS) under the general depth cap.

Reproduction

  1. Generate poc/make-nested-tables.py (~150 levels) → render → hang pre-fix (>20s), ~0.9s post-fix.

Example files:

  • poc/make-nested-tables.py

Proposed fix

Memoization alone does not fix the problem (the width computation is already cached and linear; the exponential cost comes from layout, not the width computation, and memoizing a full layout -- positions, overflow, fragmentation -- would be far more invasive and risky than a width). Chosen fix (a fallback explicitly documented in this folder as an acceptable alternative): specifically bound table nesting depth (distinct from the general cap of 512). TableBox now knows its depth among its table ancestors (m_nestingDepth, derived in build() -- amortized O(1), since the nearest ancestor already has its own depth computed, as build() walks the tree top-down). Beyond EngineLimits::maxTableNestingDepth(), TableSectionBox::layoutRows() skips the second (stretch) pass for that table's cells: the cell keeps the natural height measured by the first pass -- same content, same width, same position, just not stretched if a sibling cell in the same row is taller. Because this doubling is recursive, the total cost is O(2^limit × total_depth), not just O(2^limit) (each level past the limit re-doubles everything nested above it): the limit must therefore stay low so that 2^limit remains a small constant, unlike the other limits in this table where a value around 100 would suffice.

Configuration (default + configurable)

New EngineLimits::maxTableNestingDepth limit (setMaxTableNestingDepth/maxTableNestingDepth()), default 8, 0 = unlimited (not recommended, cost is genuinely exponential without it). C API plutobook_set_max_table_nesting_depth(unsigned int). Verified: scaling is now linear (0.19s/0.63s/0.89s/0.98s for 50/100/150/200 levels, vs. a hang/timeout pre-fix around N≈20-25); the 150-level PoC -> ~0.92s, valid PDF (pre-fix: timeout >20s). Non-regression verified by byte-for-byte PDF comparison between a pre-fix binary (git stash + a separate build) and post-fix on: a simple grid, colspan/rowspan, 2-level nesting, %/fixed/auto widths, caption (all 5 -- byte-identical); real nesting up to exactly the limit (depth 8, included) -- byte-identical; past the limit (depth 9+), differences appear only when a row contains cells of different natural heights (the shorter one is no longer stretched) -- exactly the documented behavior, no corruption.

V05 · Image decompression bomb (no pixel budget)

High   Denial of service (memory)

Status open
Key file source/resource/imageresource.cpp:100-193

Nature

There is no overall pixel budget. stb only bounds each axis (1<<24) and the allocation overflow (~2 GB); a PNG of a few KB at ~23000×23000 decompresses to ~2 GB of RGBA. turbojpeg accepts up to 65535² (~17 GB), webp up to 16383² (~1 GB), passed straight to cairo_image_surface_create.

Location

  • source/resource/imageresource.cpp:162-169 — stb: STBI_MAX_DIMENSIONS=1<<24, no total pixel cap
  • source/resource/imageresource.cpp:100-124 — turbojpeg: width/height unbounded going into cairo
  • source/resource/imageresource.cpp:139 — webp: config.input.width/height passed straight to cairo

Risk & impact

Impact: memory exhaustion from a tiny image (DoS).

Reproduction

  1. Generate the bomb: python3 poc/make-bomb.py (creates bomb.png, ~23000x23000, a few KB).
  2. Render poc/repro.html, which displays it → ~2 GB allocated.

Example files:

  • poc/repro.html
  • poc/make-bomb.py

Proposed fix

Pixel cap applied before cairo_image_surface_create on all paths (stb/turbojpeg/webp); lower STBI_MAX_DIMENSIONS.

Configuration (default + configurable)

setMaxImagePixels (default ~64 MP), configurable.

V19 · Heap::concatenateString O(n²) on text accumulation

High   Denial of service (memory)

Status fixed
Key file source/heapstring.h

Nature

TextNode::appendData recopies the entire accumulated string on every character-data callback, into the monotonic PMR arena (never freed). O(n²) growth in both time and memory. A deep XML entity bomb (or a large, fragmented text node) triggers a std::bad_alloc before even expat's amplification protection (V16) kicks in.

Location

  • source/heapstring.h — Heap::concatenateString: recopies the whole string on every call
  • source/document.cpp — TextNode::appendData calls concatenateString on every character-data callback

Risk & impact

Impact: memory exhaustion (DoS) from a small XML document.

Reproduction

  1. Generate poc/make-entity-bomb.py (deep entities) → render → bad_alloc pre-fix, bounded post-fix.

Example files:

  • poc/make-entity-bomb.py

Proposed fix

Avoid the O(n²) recopy (efficient text accumulation) and/or cap the accumulated length of a text node (sane default, configurable).

Configuration (default + configurable)

TextNode::appendData now accumulates fragments in a plain std::string (geometric growth on the normal heap, freed on each reallocation), materialized into the arena (Heap::concatenateString) only once, on first read (data()) -- O(n) cost instead of O(n²), identical text. Defense in depth: EngineLimits::maxTextNodeLength (default 100,000,000 characters, configurable, 0 = unlimited) caps the accumulated length of a single text node.
